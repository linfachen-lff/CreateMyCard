# 布局系统

当字体不合适、组件重叠、内容碰撞、留白过多、布局不合理，或紧凑布局决策不明确时读取本文档。本文只用于内部规划，不要输出布局计划对象。

## 先算这些（必须先过）

- root：先确定尺寸、安全区和内容区，`2x2` 内容区约 `116 x 116vp`，`2x4` 内容区约 `276 x 116vp`。
- 横向：每个 Row 的固定宽度、文本估算宽度、子项左右 `margin`、padding 和 `itemMargin` 相加后，不超过父容器宽度。
- 纵向：每个 Column 的元素高度、文本行高预算、子项上下 `margin`、padding 和 `itemMargin` 相加后，不超过父容器高度。
- 文本：每个受保护文本都有字体大小、行数、宽度和完整显示策略。
- 动作：每个 Button 有不小于 `24vp` 的点击视觉尺寸；和文本并排时，父 Row、Text、Button 都要有明确宽高预算。
- 层叠：每个 Stack 都能说明背景层、信息层和点击层，且不会覆盖受保护文本、CTA 或主数值。
- 对齐：同一功能组的背板、文本、按钮共享边界；堆叠元素之间仍保留 4-12vp 间距。
- 底部：支撑/动作区外框锚定安全区底边，没有漂浮；按钮和相邻文本视觉基线一致。

如果任何一项算不清，先删减弱信息或换更简单布局，再考虑升级尺寸；不要用省略、裁剪、默认伸缩或 Stack 覆盖来补救。

## 画布、尺寸和区域

- 基础尺寸、root 圆角、clip 和安全区以 `core-rules.md` 为准；本文件只补布局判断。
- root、主区域、背板、按钮和进度图形尽量写稳定 `width` / `height` / `constraintSize`，不要让动态文本自由撑开画布。
- `2x2` 最多 3 个主区域和 1 个显式动作；推荐一个 hero 区、一个支撑行、可选一个动作。
- `2x4` 最多 4 个主区域和 2 个动作区；常用横向分区是 `176-196vp` 主内容 + `68-88vp` 动作/状态 + `8-12vp` 间距，或两个 `132vp` 左右的对等区。
- 未指定尺寸先尝试 `2x2`；升级 `2x4` 必须有具体失败点：受保护文本/热区放不下、核心内容需要并列理解、关键媒体必须同屏，或用户明确要求横版/并排/双面板。
- 用户指定 `2x2` 时先删弱信息、减少区域、收敛动作；指定 `2x4` 时保持一个主对象，增加真实支撑信息、可读宽度或更清晰动作，不简单放大 `2x2`。
- 请求 `1x2`、`4x4` 或其它未声明尺寸时，不生成该尺寸；改给最接近的 `2x2` 或 `2x4` 替代方案。
- 使用固定比例带，不要临场发明比例：`62/38`、`55/45`、`50/50`、`30/70`、`15/85`。数字/状态常用 `55/45`，时间线常用 `30/70`，媒体常用 `62/38`。
- 信息文本、图标、按钮、数据图形、highlight 和背板必须在安全区内。背景图片或渐变可以贴边，但不能承载必须读取的信息。
- 固定间距尺度：只使用 `2/4/6/8/10/12/14/16`；优先 `4/8/12/16`，`2` 只做微调，`10/14` 只做严密预算修正。
- 间距默认：`2x2` 组内 4-8vp、组间 8-12vp；`2x4` 组内 6-8vp、组间 8-12vp；背板内 padding 8-10vp。不要套多层内部卡片。
- 先定义标题/状态组、主指标组、支撑信息组、动作组；同组内部间距小，跨组间距大。
- 内部半透明块用 14-18 圆角，小图标块用 5-8，pill/action bar 取 height/2；浅色卡只用低 alpha 黑色阴影，例如 `#22000000`。
- 默认左对齐。只有内容很少、单一主对象明确时，才使用居中或视觉居中。

## 预算顺序

写 JSON 前按这个顺序过一遍：

1. 定 root 尺寸、安全区和内容区。
2. 定主区域、支撑区域、动作区域的固定宽高；`2x2` 通常只允许三段，`2x4` 通常只允许左右两列或三段横向组。
3. 先给图标、进度、按钮等固定元素占位，再把剩余宽度分给文本。
4. 按文字估算规则确认受保护文本能放下；放不下先删可选信息，再降字体或换尺寸。
5. 确认所有 gap、padding、margin、itemMargin 后，横向总和不超过父宽，纵向总和不超过父高。

常用安全槽位：

- `2x2 top-hero-bottom`：顶部 18-22vp，间距 4-6vp，hero 48-58vp，间距 6-8vp，底部 26-32vp，总高不超过 116vp。
- `2x2 icon-fact-action`：图标/环 56-72vp，事实区 28-38vp，动作 28-34vp；三者不能都取上限。
- `2x4 split-action`：主内容 176-196vp，动作/状态 68-88vp，中间 8-12vp；左右列都要写稳定宽度。
- `2x4 media-info`：媒体/主视觉 92-120vp，信息列 144-168vp，动作区可选 60-76vp；媒体不能挤压标题和 CTA。

## 字体阶梯

- 只使用批准阶梯：`10fp`、`12fp`、`14fp`、`16fp`、`18fp`、`20fp`、`32fp`、`40fp`。
- hint / 占位式提示：`10fp`，只用于非主信息、刻度、弱提示；不要用作主信息。
- 支撑信息和 metadata：`12fp`，字重 Regular/Medium。
- 紧凑标题、普通正文、短状态：`14fp`，字重 Regular/Medium；重要标题可用 Semibold。
- 主标题/正文：`16fp` 或 `18fp`，只在空间足够时使用。
- 紧凑数值：`20fp`；重要数值：`32fp`；单一绝对主数值：`40fp`。
- CTA：`14fp Medium`，文字保持一行，优先 2-6 个中文字符。
- 同一卡片只保留一个最强字重；不要把所有标题、数值、按钮都设成 Bold。
- 预算垂直高度时按 `fontSize + 2-4fp` 计算。不要为了行高去输出组件目录未列出的属性。
- 系统大字体适配时，`20fp` 以上主数值不继续放大；普通文本放大上限约 `1.15x-1.3x`。

## 文本适配

- 受保护文本必须完整显示：标题、日期、星期、时间、CTA、状态、主指标、倒计时、价格/数量和入选用户字段。
- 写 JSON 前先估算宽度：`available = parentWidth - horizontalPadding - gaps - fixedIconOrDividerWidths`。
- 估算文本宽度时，中文字符约等于 `fontSize`，英文/数字约等于 `0.6 * fontSize`，空格和标点约等于 `0.3-0.5 * fontSize`。
- 受保护文本放不下时，按顺序处理：缩短周边标签，删掉可选字段，换成两行且预留高度，降低到同阶梯下限，或选择 `2x4`。
- 不要依赖 `ellipsis`、`clip` 或 marquee 隐藏受保护文本。可选文本与其截断，不如删除或移动到详情。
- 日期/时间、状态、CTA 不要同时挤在一个窄 Row 内，除非每个元素都有明确宽度预算。

## 重叠防线

- 优先使用 Row/Column 正常流；只有背景层、进度环居中、角标或明确叠加时才使用 Stack。
- Stack 中背景放底层，内容放上层；信息层必须有稳定宽高、对齐方式和 padding，不能靠视觉猜测避让。
- Stack 叠加不得覆盖受保护文本、CTA、点击视觉元素或进度主值。
- 固定高度/宽度容器内的弹性子项必须可收缩或被约束；在 DSL 中用明确 `width`、`height`、`constraintSize`、`flexShrink` 或容器 `clip` 兜底，避免内容压力把元素推出安全区。
- 使用 `spaceBetween` 时，左右两侧都要有可承受的宽度；动态长文本不要和固定按钮抢同一行。
- 重复项必须有稳定高度和间距；`List` 只用于少量真实重复项，避免滚动和密集列表。
- 每个按钮或可点击视觉元素不小于 `24vp`；周围布局尽量给到接近 `40vp` 的视觉热区。
- 同一功能组内，内容背板、文本块、按钮要共享左右边界。堆叠按钮直接服务某个内容背板时，默认和背板等宽，不要窄按钮挂在宽背板下方。

## 图标布局

- 使用素材库图标时，必须显式设置 `width`、`height` 和 `objectFit: "contain"`；不要让图片自由撑开 Row/Column。
- 辅助语义图标通常 16-24vp；主视觉图标在 `2x2` 通常 44-64vp，在 `2x4` 通常 48-72vp。
- 图标 + 文字同 Row 时，先扣除图标宽度和 4-8vp gap，再估算文本宽度；不要把长标题和固定图标硬塞同一行。
- 图标只服务识别或状态，不作为填空装饰。若加入图标会导致 CTA、时间、标题或主指标截断，放弃图标。
- 同组多个小图标必须尺寸一致、基线一致；不要在紧凑卡片中混用 16vp、24vp、40vp 多级小图标。

## 布局原型

- `hero-top`：适合 `2x2`。上方标题/状态 18-24vp，中间 hero 32-44vp，下方支撑或动作 22-32vp。
- `hero-left`：适合 `2x4`。左侧视觉/指标 72-96vp，右侧放 2-3 条事实；不要两侧都做 hero。
- `split-action`：适合 `2x4`。主内容 176-196vp，动作/状态 68-88vp，中间 8-12vp 间距。
- `paper-panel`：中性 root + 一个内容背板，适合日程、备忘、票券、账单、关怀、物流；背板 padding 8-10vp，不要再嵌套卡片。
- `meter-focus`：一个进度/仪表视觉 + 紧凑标签。进度环在 `2x2` 用 56-72vp，在 `2x4` 用 64-88vp；数值不要塞进很小的环洞。
- `ambient-root`：root 承载天气、睡眠、音乐或环境氛围；氛围可以大面积，但必须给主文本留出清晰安全区。

## 2x2 场景启发

这些是从常见 2x2 场景样例中抽象出的内部构图规则。只借鉴信息密度和区域关系；不要复制旧尺寸、超阶梯字号、emoji、SVG、data URI、多次 `updateComponents` 或不在协议内的动作写法。

- `top-hero-pill`：顶部 18-24vp 放对象/状态，中部 44-64vp 放主数值、主图标或标题，底部 28-34vp 放一个动作或支撑胶囊。适合天气、会议、专注、提醒。
- `meter-plus-fact`：56-72vp 进度环/仪表作主焦点，旁边或下方放 1-2 条短事实；需要动作时底部保留 28-34vp 胶囊。适合电量、睡眠、设备、清理状态。
- `bar-two-metrics`：顶部对象 18-22vp，中部 20-28vp 横向条带/阈值，底部两个紧凑指标或一个动作。适合告警、用量、额度、剩余时间。
- `big-number-support`：一个 `32/40fp` 数字或倒计时作为绝对主答案，配一个 10-18vp 标签和一个 28-40vp 支撑背板。适合赛事倒计时、价格、步数、日程日期。
- `ambient-info`：root 用渐变或合法背景建立氛围，信息层仍在安全区内；文字必须落在清晰前景/弱遮罩上，不把噪声背景当作可读区域。

`2x2` 的可读容量上限通常是：一个标题/状态、一个 hero、一个支撑组和一个动作。若必须同时保留两个并列信息组、两个动作、长标题 + 媒体 + 详情，才考虑 `2x4`。

## 背板和动作关系

- `2x4` 右侧支撑组如果是多行散点信息、小矩阵、多个 label/value，要放入一个中性背板；不要让它们散落在 root 上。
- 单个孤立元素不需要背板：一个按钮、一个状态行、一个图标如果没有同组内容，单独包背板会增加噪声。
- 有支撑文本 + 一个动作的面板，按钮通常贴近面板底部并与内容同宽；按钮上方必须有真实内容填充。
- 如果按钮上方只有一行短文本和大片空白，放弃底部锚定：加真实支撑行、拓宽内容列，或把内容和按钮作为一个居中组。
- 底部支撑/动作区服务整张卡片时，它的外框应贴近安全区底边。`2x2` 外框底边到卡片底边通常保持 `8-12vp`，最大不得超过 `16vp`；`2x4` 通常保持 `10-14vp`，最大不得超过 `16vp`。不要只依赖 `spaceBetween` 的几何分布；必须检查截图观感，若按钮下方留白大于其上方相邻间距，或按钮看起来靠上/悬浮，则继续下移、增高动作区或重分配上方内容高度。
- 底部操作区必须与上方主内容、进度条或支撑背板共享同一左右边界；不要让按钮组比它服务的内容明显窄、偏移或单独悬挂。
- 紧凑 Row 中按钮与小号文本并排时，必须检查视觉基线；如果按钮文字看起来偏上或偏下，优先调整按钮槽位高度或上下 padding。只有父 Row 高度仍能容纳 `Button.height + margin.top/bottom` 时，才用 `margin.top` / `margin.bottom` 做 `1-2vp` 校正。

## 留白处理

- 6-12vp 是呼吸感；超过 18vp 的连续空洞必须有明确用途，例如媒体、进度图形或氛围背景。
- 如果 hero 区低于内容区面积约 40%，且只有一行小字，卡片通常会显得空。放大 hero、加入一个有用支撑信息，或改用 `2x2`。
- sparse 内容可以让主对象占可用高度的约 `75-85%` 视觉质量，但仍必须使用批准字号、图标和间距阶梯。
- `2x4` 不要把一个短标题居中放在大画布里。使用左右分区、背板组或动作区建立横向张力。
- 不要为了填空增加无意义装饰。优先调整比例、删减区域、放大主答案。

## 输出前检查

- 间距：每个 gap、padding、itemMargin、margin 都能回溯到固定尺度；组间距大于组内距。
- 密度：没有滚动、长列表、表格、按钮网格、密集多面板或无意义大留白。
