# 可泛化卡片构造规则

本文档替代基于模板的生成。它定义 `2x2` 与 `2x4` HarmonyOS A2UI Form 卡片的可复用构造规则，并集中承载生成前和生成中的构图/设计决策。

## 构造心法

从用户含义出发，而不是从示例产物出发。

1. 提取语义角色。
2. 选择 `2x2` 或 `2x4`。
3. 分配尺寸预算。
4. 用简单语法组合区域。
5. 按语义绑定数据。
6. 生成草稿，再评审视觉质量。

## 卡片边界

- Form 卡片是一眼可读的摘要容器，不是迷你页面。
- Form 子集不支持 `Card` 组件；使用带样式的 `Column`、`Row` 或 `Stack` 作为唯一卡片 shell。
- Root shell 是唯一卡片承载层；背景可延展到边缘，内容必须避开宿主圆角裁剪区。
- 默认避免长列表、长表格、长时间线、多段解释、多个大型并列区块和完整文章/详情页结构。
- 如果核心内容无法在 `2x2` 或 `2x4` 内无滚动容纳，使用能力边界说明，不要悄悄生成页面尺寸卡片。

## 基础设计约束

- 布局优先追求秩序、整洁和特征性：信息有顺序、对齐稳定、阅读路径清晰。
- 固定布局默认左对齐；特殊布局也必须保持基础对齐、留白和间距节奏。
- 内容层距离卡片四周至少 `12vp`。背景图、渐变或装饰层可以贴边，但不能承载必须完整读取的信息。
- 不要把主标题、主数字、状态、CTA 或可点击控件放进最终圆角裁剪可能影响的角落区域。
- 除核心信息外不要追加解释性内容；用品牌色、品牌纹理或特征图形时，只作为场景特征和层级辅助。

## 信息摘要和缺失数据

当一句话请求是实体、概念、地点或事件摘要时：

- 使用 `title -> 1 到 3 个属性 -> <= 2 行短摘要 -> 可选动作`。
- 默认不要使用动态列表模式。
- 不要使用全宽 16:9 hero 图；这是页面模式。
- 如果用户需要更多详情，建议页面范围，而不是扩大卡片。

从稀疏用户输入生成时：

- 仅在自包含演示需要时，推断中性但相关的展示值。
- 按语义分组 DataModel，例如 `weather`、`meeting`、`action`、`asset`、`progress`。
- 不要留下示例或历史输出中的样例值。
- 缺少语义匹配的素材库素材、用户提供图片或真实本地资源时，省略 `Image`，使用非媒体视觉技法。若素材库有明确匹配且该语义视觉锚点仍需要展示，必须使用素材库 `src`，不要改用自造图标或文本字形。

## 媒体和表现力选择

- `Image.src` 只使用用户提供或 [`asset-library.md`](asset-library.md) 声明的本地/资源路径，不使用网络 URL、SVG 或未声明资源路径。
- 需要语义图标/图片时，先匹配素材库；匹配成功后必须用对应 `src`，不得使用 `Text` 字形、emoji、自绘形状或相似路径替代。
- 图片背景卡使用 `Stack` root，`Image` 作为第一个 child，内容叠层作为第二个 child。
- 背景/hero 填充使用 `styles.objectFit: "cover"`。
- 如果图片不能增加场景特定信息，避免使用图片。
- 有语义匹配的素材库素材、用户提供图片或真实视觉素材时使用 `Image`。
- 进度/环形/胶囊指示使用 `Progress`。
- 轻量结构使用 `Divider`。
- 层叠光晕、图片或进度组合使用 `Stack`。
- 重复紧凑事实谨慎使用 `List`；Form 不支持 `Grid`。
- 当最稳定时，使用普通 `Text` 承载标签、数值和类似图标的字形。

## 语义角色

每张生成卡片在写 DSL 前，都应把用户内容分配到这些角色：

| 角色 | 含义 | 常用组件 |
| --- | --- | --- |
| `identity` | 这张卡片关于什么：标题、设备名、人物、地点、事件 | `Text`、小 `Row` |
| `primaryAnswer` | 最重要的一眼可见结果 | 大 `Text`、`Progress`、图片焦点 |
| `metric` | 数字、时间、温度、电量、价格、时长 | 受保护 `Text`、`Progress` |
| `context` | 次要说明或临近事件/详情 | 紧凑 `Column`、`Row`、`Divider` |
| `status` | 良好/警告/错误/模式标签 | 带背景、字形或强调色的小 `Text` |
| `media` | 用户提供或素材库声明的真实本地/资源素材 | `Stack` 内的 `Image` |
| `action` | 一个下一步操作 | 可点击 `Stack`/`Row`/`Column` 或 `Button` |

规则：

- `primaryAnswer` 必须在视觉上占主导。
- `context` 必须短于 `primaryAnswer`。
- `action` 支持卡片，不应成为主视觉，除非用户要求纯动作卡。
- 如果没有语义匹配且可靠的 `media`，使用渐变、半透明块、字形文本、`Divider` 或 `Progress`。字形文本只用于无匹配素材的简单符号，不用于替代已匹配素材。

## 尺寸决策规则

先把请求压缩成 `primaryAnswer + context? + action?`。如果能在 `2x2` 中形成 2 到 3 个主区域，就不要升级。选择 `2x4` 前，必须先写出 `2x2` 的具体失败点。

使用 `2x2` 当：

- 卡片有一个主答案，最多一个上下文组。
- 用户要求紧凑 widget 或单一状态/提醒。
- 单一主题只需要 1 到 2 个短指标/状态、一个短 CTA 或一个产品/图标视觉锚点。
- 睡眠状态、会议提醒、关怀/联系、倒计时 + 单条计划、天气 + 一个事件、低电/连接状态、防沉迷时长等场景都应先判为 `2x2`。
- `160 x 160vp` 布局可以无滚动容纳。

使用 `2x4` 当：

- `2x2` 经过字段筛选、减 padding/gap、降字号、拆行后仍无法完整显示受保护关键信息。
- 卡片需要 2 到 3 个必须并列理解的独立 pane；这些 pane 不是一个主信息下的短上下文。
- 媒体本身承载关键信息，且必须与状态、详情和动作同时可见；纯装饰图片或单个产品视觉锚点不算。
- 用户明确要求横版、并排对比、双面板、左右关系或 3 个以上紧凑辅助事实。

不要因为“媒体加状态”“详情加动作”“关怀状态加联系”“设备/产品状态”这些场景名本身升级；只有当内容容量或热区预算证明 `2x2` 不够时才升级。

以下情况不要用 `2x4`，而应升级说明：

- 需要超过 4 个主要区域。
- 内容需要段落、表格、表单或长动态列表。
- 用户真正想要详情页。

## `2x2` 规则：`160 x 160vp`

### 硬预算

- Root 目标：`width: 160`，`height: 160`。
- 内容安全边距：`12`。
- 外层圆角预览：`18vp`。
- 可用内部区域：约 `136 x 136vp`。
- 主区域：<= 3。
- 直接 `Row` 子节点：<= 3。
- 不滚动。
- 默认不使用动态列表模式。

### 区域预算

选择一种分配节奏：

| 节奏 | 使用场景 | 预算 |
| --- | --- | --- |
| `header-primary-action` | 提醒、事件、CTA 卡 | header `18-24`，primary `58-72`，action `30-36` |
| `primary-context-action` | 带详情的状态/指标 | primary `58-76`，context `34-44`，action `28-34` |
| `visual-primary-context` | 进度/设备/健康速览 | visual `56-76`，primary/context 分享剩余高度 |
| `single-focal` | 一个强结果，可选极小 meta | focal `88-110`，meta/action `22-34` |

这些预算是指导，不是精确模板。用它们避免文本在 `160vp` 内互相碰撞。

### 组合语法

使用一个 root shell：

```text
RootShell(Column|Stack)
  -> Identity?
  -> PrimaryAnchor
  -> ContextStrip?
  -> ActionPill?
```

允许的变化：

- 使用图片背景、光晕或进度叠层时，root 可用 `Stack`。
- 以文本/指标/动作为主时，root 可用 `Column`。
- 只有当区域内部表达真实关系时才用 `Row`：图标 + 标签、指标 + 状态、进度 + 数值。

### 文本预算

- 标题/identity：1 行。
- 主值：1 行；如果值可能变化，使用 `minFontSize`。
- 上下文：1 行；如果没有动作，可用 2 行非常短的文本。
- CTA：1 行；按内容宽度或全宽 pill。
- 避免任何像段落的正文。

### 视觉规则

- 只保留一个视觉锚点：大指标、进度环、图片主体、图标/字形或状态徽章。
- 最多 1 个主要半透明块。
- 只有当图片真实且有用时才使用图片。否则使用非媒体视觉技法。
- 保持强对比；图片上的文字需要遮罩或安全位置。

## `2x4` 规则：`160vp（高）x 320vp（宽）`

### 硬预算

- Root 目标：`width: 320`，`height: 160`（即 `160vp（高）x 320vp（宽）`）。
- 内容安全边距：`12`。
- 外层圆角预览：`22vp`。
- 可用内部区域：约 `136vp（高）x 296vp（宽）`。
- 主区域：<= 4。
- 直接 `Row` 子节点：<= 3。
- 默认不滚动。如果静态内容放不下，收敛内容。

### 区域预算

选择一种横向节奏：

| 节奏 | 使用场景 | 预算 |
| --- | --- | --- |
| `focal-detail-action` | 通用丰富紧凑卡 | focal pane `108-132w`，detail pane `120-150w`，action/status `40-56w` 或底部条 |
| `media-state-action` | 带真实图片的产品/设备/地点 | media pane `112-140w`，state/details `120-150w`，action `40-56w` |
| `metric-context-action` | 天气、进度、健康、财务 | metric pane `96-124w`，context pane `132-164w`，action/status `44-60w` |
| `sequence-detail-action` | 日程/计划/关怀卡 | sequence pane `132-164w`，detail pane `92-124w`，action/status `44-60w` |

2x4 卡片可以有两个并排 2x2 区域的感觉，但必须仍是一个卡片 shell 和一个层级。

### 组合语法

```text
RootShell(Row|Stack)
  -> FocalPane
  -> DetailPane
  -> ActionOrStatus?
```

允许的变化：

- 每个 pane 内使用 `Column` 形成竖向节奏。
- 当真实图片/背景或光晕层跨越整张卡时，root 使用 `Stack`。
- 只有在受保护文本仍可读时，才把 `ActionOrStatus` 放为右侧栏、底部条或最后一个详情行。
- 使用 2 到 3 行静态详情簇；除非用户明确需要可变重复数据，否则避免动态列表。

### 文本预算

- Focal pane：1 行主值/标题，加 1 个可选 caption。
- Detail pane：2 到 3 个紧凑行，每行 1 行。
- 建议/正文：总计 <= 2 行短文本。
- CTA/status：1 行，受保护；除非已知完整标签能放下，否则避免窄固定 CTA。

### 视觉规则

- 一个主导视觉锚点，最多一个次级视觉装置。
- 最多 2 个主要半透明块。
- 进度环、媒体主体或大数字可以占据左 pane。
- 右 pane 应承载详情和动作，不要再放一个竞争性的 hero。

## 场景到结构映射

这些是规则，不是模板。先按语义角色选择结构，再为场景创建新的 ID、数据键、颜色和间距。

| 场景 | 优先尺寸 | 结构逻辑 |
| --- | --- | --- |
| 天气 + 下一件事 | `2x2` | 主天气指标 + 下个事件上下文 + 动作/状态 |
| 会议/提醒 | `2x2` | identity 标签 + 受保护时间/标题 + 地点/动作 |
| 倒计时 | 单个事件或事件 + 单条计划用 `2x2`；多条计划/趋势/动作并列才用 `2x4` | 大倒计时 + 单位 + 辅助计划/动作 |
| 健康/进度 | 单指标或指标 + 单条建议用 `2x2`；指标 + 趋势 + 多字段建议才用 `2x4` | 进度视觉 + 时长/数值 + 提醒/建议 |
| 家庭关怀告警 | 默认 `2x2`；只有风险详情、天气/位置和联系动作都必须并列保护时用 `2x4` | 状态/天气 + 风险详情 + 联系动作 |
| 设备/产品状态 | 产品/图标 + 1 到 2 个状态 + 1 个动作默认 `2x2`；宽媒体 + 3 个以上状态/控制才用 `2x4` | 真实素材或字形 + 电量/模式行 + 动作 |
| 信息摘要 | 标题 + 1 到 2 个属性 + 一句短上下文用 `2x2`；3 个以上属性或两行摘要 + 动作无法容纳时用 `2x4` | 标题 + 属性 + <=2 行摘要 + 动作 |

## 宽度与换行规则

宽度预算严格：`2x2` 宽 `160vp`，横版 `2x4` 宽 `320vp`。两者高度都只有 `160vp`。

- 只有在值长度已知且完整字符串能放下时，才给受保护值设置显式宽度。
- 文本列紧邻固定图标或仪表时，应使用 `layoutWeight: 1`。
- 如果一行同时包含指标和句子，句子是可压缩内容。
- 对受约束的关键文本使用 `maxLines: 1` 和 `minFontSize`。`textOverflow: "ellipsis"` 只用于可选次要文本。
- 不要把短中文标签、单位、时间、价格、温度或 CTA 标签按字符拆开。

## 字号与字重规则

卡片展示空间小，字号用于区分层级，不用于制造页面感。

| 用途 | 字号 |
| --- | --- |
| 超大数字 | `40fp` |
| 大号数字 | `32fp` |
| 中号数字 | `20fp` |
| 标题 | `18fp` 或 `16fp` |
| 正文/辅助信息 | `16fp`、`14fp` 或 `12fp` |
| 提示性文本 | `10fp` |

规则：

- 最小 `10fp` 仅用于数据刻度、提示性文字或极弱辅助信息，不作为主要信息展示。
- 合理使用 `Regular`、`Medium`、`Semi-bold`、`Bold` 建立筛选关系；不要整卡都使用粗体。
- `2x2` 中标题通常使用 `14fp` 或 `18fp`，副标题/辅助信息通常使用 `12fp`；主数字可使用 `32fp` 或 `40fp`，辅助数字使用 `12fp` 或 `14fp`。
- `2x4` 仍遵守 `160vp` 高度节奏；横向空间用于增加关系和受保护宽度，不用于继续放大字号。
- 左上角标题默认从内容安全边距开始，使用 `14fp` 或 `18fp`。

## 关键信息完整显示规则

除非用户明确接受截断，否则关键信息必须完整可见。

默认把这些视为关键信息：

- 日期、星期、时间、倒计时、时长、温度、价格、评分、电量、百分比和主指标值。
- CTA 标签、状态徽章、告警标签、短中文短语和主事件/物品标题。
- 字段筛选后决定展示的用户字段；当用户需求涉及太多数据字段时，最多保护 4 项，未入选字段不列为受保护内容。

规则：

- 不要在关键 `Text` 组件上设置 `textOverflow: "ellipsis"`。
- 不要把 ellipsis 改成 `clip` 来修关键文本；裁剪仍然会隐藏信息。
- 如果关键文本横向放不下，写 JSON 前先重新平衡布局：
  1. 减少装饰性 padding 和 `itemMargin`。
  2. 减少固定列和分隔线厚度/宽度。
  3. 在保持层级的前提下降低字号。
  4. 把可压缩次要文本移动到另一行，或从演示数据中移除。
  5. 把该行拆成 `Column`、选择横版 `2x4`，或在请求无法容纳时升级说明。

## Row 宽度预算

生成任何拥挤 Row 前，在布局理由中写出计算：

```text
available = parentWidth - horizontalPadding - totalItemMargins - fixedIconOrDividerWidths - fixedTextWidths
```

然后检查每个受保护字符串能否放入分配宽度。对 `2x2` 卡片，按 `12vp` 内容安全边距后父级内容宽度通常为 `136vp`。在嵌套半透明块内部，padding 和行间距很容易让可用宽度低于 `120vp`；当多个受保护字符串共用一行时，分隔线和间距要小。

对于必须在 `2x2` 卡片中显示时间 + 标题 + meta 的紧凑提醒/事件行：

- 事件块水平 padding 保持在约 `4-6vp`。
- Row 的 `itemMargin` 保持在约 `3-4vp`。
- 使用细分隔线（`strokeWidth: 1`），或在造成拥挤时省略。
- 当标题和 meta 也必须显示时，`HH:MM` 时间保持约 `18fp`、`48-50vp` 宽，而不是大号 `22fp`/`56vp` 块。
- 同一个紧凑块内共享空间时，短中文标题约 `12fp`，meta 约 `9fp`。

对于横版 `2x4`，按 `12vp` 内容安全边距后父级内容宽度通常约 `296vp`。优先使用 2 或 3 个 pane 的主 `Row`，并明确预算每个 pane。不要把额外宽度浪费在过大的装饰间距上；用它保证受保护值完整。

## 按尺寸的交互规则

`2x2`：

- 通常一个主可点击区域或一个全宽动作 pill。
- 避免超过一个 CTA。

`2x4`：

- 一个主动作加一个可选被动状态是可以接受的。
- 如果有多个动作，它们必须视觉上从属、由用户明确要求，并且能放进横向右侧栏或底部条而不被裁剪。

## DataModel 形状

按含义分组数据，而不是按布局分组：

```json
{
  "primary": {"label": "Status", "valueLabel": "31 deg"},
  "context": {"title": "Next meeting", "meta": "09:30 A14"},
  "action": {"label": "Open", "targetId": "demo-001"}
}
```

规则：

- 不要用 `topText1` 这种只反映视觉区域的数据键。
- 当格式化或本地化重要时，预先计算展示字符串。
- 从 DataModel 绑定动作 ID，不要在事件处理器中硬编码业务 ID。

## 生成前自检

写 JSON 前：

- [ ] 我是否明确选择了 `2x2` 或 `2x4`？
- [ ] 我是否识别了语义角色？
- [ ] 结构是否能在尺寸预算内无滚动容纳？
- [ ] 是否只有一个主导焦点？
- [ ] 受保护值是否不会被窄宽度换行破坏？
- [ ] 我是在按规则构造，而不是改造模板吗？
