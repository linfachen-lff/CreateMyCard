# 设计启发式

只有视觉质量偏弱、用户要求润色、卡片看起来过于泛化，或需要决定 composition、表面策略、渐变和表现技法时读取本文档。本文不能替代 [`layout-system.md`](layout-system.md) 的尺寸预算和 [`color-token-system.md`](color-token-system.md) 的色值校验。

## 使用顺序

1. 先确定一个主答案和必须保留的用户字段。
2. 再选 composition pattern 和 surface strategy。
3. 再给颜色角色和状态色分工。
4. 最后才使用渐变、半透明块、文本字形或图片背景等表现技法。

如果表现技法压缩了受保护文本、CTA、日期、时间、状态或主指标，放弃该技法。

## 紧凑卡片原则

- 让用户在 1-2 秒内看到一个主答案。
- `2x2` 要克制：一个主焦点、一个支撑细节、可选一个动作。
- `2x4` 可以增加一个支撑组或第二动作区，但仍然只能有一个主焦点。
- 用户未指定尺寸时，不要默认升级 `2x4`；先判断 `2x2` 是否能放下一个主答案、1-2 条支撑和一个动作。
- 长描述、额外字段、第二行解释和第三层元信息不要塞进卡片。
- 日期、时间、状态、CTA、标题、主指标、倒计时、价格/数量和入选用户字段必须完整显示。

## 内容取舍

- 先用一句话确定用户 1-2 秒内应看到的主答案，后续尺寸、布局、色彩和动作都围绕它服务。
- 必保留：主对象、主状态/数值、主时间、主动作、关键单位；没有它卡片无法回答用户。
- 可保留：短地点、短状态、一个功能图标、1 条支撑信息；空间允许才保留。
- 移出卡片：长描述、规则解释、第三层元信息、额外动作、营销入口、长列表、表格。
- 字段筛选后最多保护 4 项用户字段；未入选字段不要作为受保护内容强行塞入卡片。
- 合理 mock 数据可以支撑布局，但不能被当成真实业务 ID、隐私数据、事件参数或素材路径。

## Composition 与表面策略

写 DSL 前只在内部确认：场景锚点是否具体、composition pattern 是否匹配内容类型、表面策略是否清晰、色彩角色是否各司其职、渐变是否必要且合法、状态色是否只服务真实状态、当前尺寸的色彩预算是否超标。

`materialAnchors` 是 2-4 个具体物理或文化锚点，例如雨玻璃、日历纸、跑道、票纸、电池环、包裹标签、药盒、收据纸或陶瓷盘。不要使用高级、现代、清新、可信、科技感、重要、漂亮这类抽象锚点。

Composition patterns：

- `plain-card`：简单工具型信息。
- `paper-panel`：会议、日历、票券、备忘、关怀、账单、物流。
- `ambient-root`：天气、睡眠、音乐、环境状态。
- `dark-focus`：夜间、演出、睡眠、专注。
- `meter-dashboard`：一个仪表或进度主焦点，不是复杂仪表盘。
- `split-action`：内容和动作有清晰分区。
- `media-card`：真实媒体或图片承载主对象。

Surface strategy：

- `plain`：中性 root 和文字层级。
- `tinted-surface`：中性 root + 同族弱色支撑背板。
- `colored-root`：root 承载场景色；前景使用 on-color 或中性 token。
- `split-surface`：主区和支撑区有清晰表面关系。
- `image-derived`：真实图片主导；文本放在安全遮罩或中性背板上。
- `dark-stage`：暗色表面有语义价值。
- `soft-material`：轻微官方材质或弱线性渐变。

## 色彩角色

- `cardSurface`：root 或主表面；承载整体氛围，优先来自官方 token、官方多彩色或真实图片；当场景需要更细腻表面时，可使用 `color-token-system.md` 的场景拓展色板，DSL 中写对应 hex 色值。
- `contentSurface`：中性支撑背板，承载次要信息。
- `sceneAccent`：唯一主场景色族。
- `actionMaterial`：默认中性或磨砂；只有动作颜色本身有语义时才实色。
- `statusAccent`：`confirm`、`warning` 或 `alert`；只服务真实状态。
- `tokenSources`：非图片颜色使用的 token 名、`ohos_id_color_*`、`multi_color_*`、`multi_color_aux_*` 或场景拓展色来源，以及最终写入 DSL 的 hex 色值。

色彩预算：

- 每张卡只使用一个主场景色族；场景拓展色必须从这个色族或中性背景族派生。
- 次要信息使用中性背板。
- `2x2`：一个场景色信号 + 可选状态/动作信号 + 中性文字。
- `2x4`：一个场景色信号 + 一个支撑表面 + 可选状态/动作信号。
- 不要让每个内容块都有自己的主题色。

状态色：

- 默认不使用状态色。
- `confirm`：完成、已连接、可用、安全、已检查。
- `warning`：真实风险、紧急临近项、缺失动作、需要注意。
- `alert`：破坏性、失败、取消、严重警告。
- 普通提醒和轻建议用层级、文字和位置表达，不使用 `warning` / `alert`。
- 状态色必须有文字、图标或位置辅助，不能只靠颜色表达。

按钮和渐变：

- 按钮默认中性或磨砂；只有确认、连接、清理、危险、真实品牌动作等语义动作才用实色。
- 渐变只使用线性渐变，stop 必须来自官方语义 token、`ohos_id_color_*`、`multi_color_*`、`multi_color_aux_*` 或合规场景拓展色，DSL 中写对应 hex 色值。
- 遇到天气、睡眠、音乐、环境、运动、倒计时等有明确氛围或动势的场景，优先尝试一个合法的弱线性渐变作为 root 或重点面；若它降低信息可读性或只是装饰，则回退为中性表面。
- `2x2` 最多一个渐变面。
- `2x4` 最多一个 root 渐变加一个中性支撑背板，或中性 root 加一个渐变重点面。
- `temporal-band` 用于 time-of-day、night、seasonal、event 或 countdown。
- `ambient-band` 用于天气、睡眠、环境或音乐氛围。
- `action-fill` 只用于运动、导航、清理、紧急等高能动作。

## 布局、字体和间距

- 保持 `12vp` 安全区。
- 使用稳定区域比例，不要自由堆叠组件。
- 先给主内容、支撑信息、动作区分配大致宽高，再写 DSL；不要让动态文本决定整体布局。
- 组件之间宁可使用 4-12vp 的稳定间距，也不要靠大块空白制造高级感。
- 避免仪表盘、按钮网格、长列表、表格和密集多面板。
- 用更少的文字层级和更明确的间距建立秩序。
- 默认字体层级控制在 3 档以内：hero、标题/状态、支撑信息。
- 只使用 `10/12/14/16/18/20/32/40fp` 字体阶梯；`10fp` 不做主信息，`40fp` 只给单一主数值。
- 数值 hero 通常用 `32fp`，单一绝对主数值可用 `40fp`，紧凑数值用 `20fp`。
- 标题/状态通常 `14/16/18fp`，支撑信息 `12fp`，CTA `14fp Medium`。
- 受保护文本不要依赖省略号。
- 正文避免默认 Bold；把强字重留给 hero 值、状态或 CTA。
- 按钮文本必须一行可读。
- 图标、按钮、背板和文本间距要稳定，动态内容不应造成布局跳动、重叠或挤压。
- 如果出现大于 `18vp` 的连续空白，检查它是否服务主视觉；否则放大主答案、重分区域或改用更小尺寸。

## 表现技法

所有技法都必须在 HarmonyOS A2UI Form 子集内完成；不要为了视觉效果新增组件目录没有声明的属性。

### 线性渐变 Shell

可在 root 或一个重点面上使用 `styles.linearGradient`：

```json
"linearGradient":{"direction":"RightBottom","colors":[["#ff202224",0],["#634794",1]]}
```

规则：

- stop 必须来自官方 token、官方多彩色表或合规场景拓展色，并在 DSL 中写对应 hex 色值。
- 只使用线性渐变。
- 不用 orb、bokeh、径向装饰或无来源 alpha 叠色。

场景拓展色适合用在 `ambient-root`、`soft-material` 和 `paper-panel`：例如天气晨间可用暖米白与薄云蓝，运动健康可用薄荷浅表面，音乐耳机可用轻品牌氛围。每张卡最多声明一个场景拓展色板，不要把多个场景的质感混在一起。

### 半透明块

使用 `backgroundColor`、`borderColor`、`borderRadius` 组织次要上下文：

```json
"styles":{"backgroundColor":"#19000000","borderWidth":1,"borderColor":"#33000000","borderRadius":14}
```

规则：

- `2x2` 最多 1 个主要半透明块。
- `2x4` 最多 2 个主要半透明块。
- 背板只用于归拢多个相关元素；单独按钮、状态或 icon 不套背板。

### 文本字形图标

当没有语义匹配素材时，可用 `Text` 承载简单字形或短符号。若素材库存在匹配素材且卡片需要该视觉锚点，不要用文本字形替代。

```json
{"id":"weatherIcon","component":"Text","content":{"path":"/weather/icon"},"styles":{"fontSize":32,"maxLines":1,"textAlign":"center"}}
```

文本字形不能是 emoji；如果宿主协议没有声明可访问性属性，用相邻可见文本承载语义，不要输出协议外字段。

### Progress、Divider 和图片背景

- 睡眠、习惯、电量、完成度、训练和目标卡可以使用 `Progress`；进度组件应是主要焦点之一，不做很小的附带装饰。
- `Progress` 颜色来自 token 系统；是否成立先按 [`layout-system.md`](layout-system.md) 的进度语义判断。
- `Divider` 只做轻量结构元素，应减少拥挤，而不是抢占受保护文本宽度。
- 图片背景只有真实素材有用时才使用；图片路径来自用户提供或 [`asset-library.md`](asset-library.md) 声明，文本放在安全遮罩或独立面板上，不遮挡重要产品或媒体细节。

## 反模式

- 同一张 `2x2` 同时使用渐变、光效、图片和多个块。
- 不能澄清场景的纯装饰。
- 多个无关强调色。
- 当有有意义的 `Progress` 或图片时，用纯文本替代视觉数据。
- 视觉表现压缩 CTA、日期、时间、标题、状态或主指标。

## 快速美观检查

输出前在内部检查：

- 是否只有一个 hero？
- 是否只有一个主色族？
- 受保护文本能否完整放下？
- Row/Column 的宽高预算是否能成立？
- Stack 是否只用于背景、进度或明确叠加，且没有覆盖文字/动作？
- 是否存在无意义的大留白？
- 每个彩色/状态元素是否都有语义作用？
- 去掉一个装饰元素后，卡片是否更清楚？
